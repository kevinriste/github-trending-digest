# Cross-Edition Navigation Design

**Date:** 2026-07-18
**Status:** Approved

## Problem

The AI Edition (third page type, added alongside GitHub Trending and Hacker News)
shipped with navigation gaps:

1. The AI magazine masthead falls into `_render_masthead`'s `else` (GH) branch:
   the own-calendar link is labeled "GitHub Calendar" (the href is correct), the
   "Hacker News" link resolves to nonexistent `docs/ai/hn/<date>/` (broken), and
   there is no link to that day's GitHub Trending.
2. The AI classic page links "GitHub Trending" to the GH *calendar* rather than
   the same-date daily page, and has no Hacker News link.
3. GH/HN pages (dailies, magazines, calendars) do not link to the AI edition at all.
4. `docs/dates.json` — the manifest `preference.js` uses for prev/next day-nav —
   only lists `gh` and `hn` dates, and has two independent writers
   (`trending_digest.write_dates_manifest` and `scripts/build_dates_manifest.py`),
   both AI-unaware. The AI magazine's day-nav therefore has no data.
5. The AI classic page does not include `preference.js`, so it lacks the style
   toggle and day-nav that GH/HN classic pages have.

Root cause: edition identity (labels, paths, read keys, date inventories) is
scattered across hand-written template strings and per-file branches, so a new
edition can silently inherit wrong labels and broken relative paths.

## Constraints

- The AI edition is **not daily**; GH and HN are.
- Import direction: `trending_digest` imports `morning_edition` at module level;
  `morning_edition` must not import `trending_digest` at module level (it uses
  deferred imports and duplicated constants today). `ai_edition` imports both.
- localStorage read-key **values** must not change (users' read-day history).
- `docs/` is the published GitHub Pages output; nothing internal goes there.

## Linking rules (agreed with user)

- From a **dated page** (classic daily or magazine): link each *other* edition's
  same-date page when it exists. If it does not exist and the target edition is
  daily (GH/HN), fall back to that edition's calendar. If the target is
  **non-daily (AI), omit the link entirely** — no calendar fallback.
- From a **calendar page**: always link the other editions' calendars.
- Existence is evaluated at generation time; no later-in-the-day look-back.
  Instead, the daily run picks up a new AI newsletter *first* (section:
  Pipeline ordering), so same-day AI links appear whenever the sidecar arrived
  before the morning run.

## Design

### 1. `editions.py` — single source of edition identity

New stdlib-only module (no project imports → safe for every module to import at
top level).

```python
@dataclass(frozen=True)
class Edition:
    id: str            # "gh" | "hn" | "ai"
    name: str          # link label: "GitHub Trending" / "Hacker News" / "AI News"
    calendar_label: str  # "GitHub Calendar" / "Hacker News Calendar" / "AI News Calendar"
    root_path: str     # path under docs root: "" / "hn/" / "ai/"
    output_dir: Path   # docs / docs/hn / docs/ai
    read_key: str      # existing gtd:read_days:* values, unchanged
    daily: bool        # gh/hn True, ai False

EDITIONS: dict[str, Edition]  # insertion order = nav display order: gh, hn, ai
```

Functions:

- `published_dates(edition_id) -> set[str]` — scan `output_dir` for ISO-date
  child dirs.
- `cross_edition_links(current_id, day_str=None, known_dates=None) -> list[tuple[str, str]]`
  — returns `(href, label)` pairs for the *other* editions, applying the linking
  rules above. All relative-path math lives here: a dated page sits at
  `<root_path><date>/` (depth = root depth + 1), a calendar at `<root_path>`.
  `known_dates: dict[str, set[str]] | None` lets callers override the
  filesystem scan per target edition (trending_digest passes its DB-derived
  gh/hn date sets, which include the in-progress day whose dirs may not exist
  yet mid-run); unlisted editions fall back to `published_dates`.

`READ_DAYS_KEY_GH/HN` in both existing modules and `morning_edition.CONFIGS`'
`output_dir`/`read_key` become references to this registry (values unchanged).

### 2. All nav blocks consume the helper

- `morning_edition._render_masthead`: drop the hardcoded `hn`/else branch.
  Own-calendar link uses `EDITIONS[config.id].calendar_label`; other-edition
  links come from `cross_edition_links(config.id, day_str)`; "Classic View"
  stays. Side effect: HN masthead label "HN Calendar" → "Hacker News Calendar".
- `trending_digest` GH/HN classic dailies + both calendar indexes: nav built
  from the helper (DB date sets passed as `known_dates`). GH and HN calendars
  gain an "AI News Calendar" link.
- `ai_edition` classic page: `← AI News Calendar · Magazine view ·
  GitHub Trending (dated) · Hacker News (dated)`. AI calendar keeps its two
  calendar links, now generated by the helper.

### 3. Pipeline ordering — pick up new AI newsletter during the daily run

At the start of `trending_digest.main`'s page-generation phase: read the
sidecar (`ai_edition.DEFAULT_SIDECAR`); if it exists and its date has no
`docs/ai/<date>/` yet, call `ai_edition.build_pages(sidecar)` via a deferred
import (the codebase's established cycle-avoidance pattern). GH/HN generation
then sees the AI date on disk and emits dated AI links. The run's existing
single commit/push covers the AI files; the links email appends the AI URL when
a new AI edition was rendered. A missing/stale sidecar is not an error.
`ai_edition.py` remains runnable standalone with identical behavior.

### 4. dates.json and preference.js

- The manifest-building logic moves into `editions.py`
  (`write_dates_manifest(docs_dir)`, deriving the edition list from `EDITIONS`,
  now including `"ai"`). `trending_digest` calls it directly and
  `scripts/build_dates_manifest.py` becomes a thin CLI over it — one writer,
  two entry points.
- AI classic page includes `preference.js` (same pattern as GH/HN classics).
  Its path-based classic/magazine sibling logic already works under `ai/`.
- `ai_edition`'s standalone publish path also rebuilds `dates.json`.
- Verify `preference.js` handles a manifest key with gaps (non-daily dates);
  prev/next is list-based, so gaps are expected to work — confirm in testing.

### 5. Verification

- New `tests/test_editions.py` (pytest, added as dev dependency — first tests
  in the repo) covering `cross_edition_links`: path math from each edition's
  dated and calendar pages, dated-vs-fallback-vs-omit rules, `known_dates`
  override.
- End-to-end: render AI pages via `ai_edition.py --no-publish` against the real
  sidecar, and regenerate a GH/HN day locally; grep the emitted hrefs/labels;
  spot-check day-nav and style toggle in a browser (Playwright) against a local
  static server.
- Historical daily pages are not regenerated: no AI editions existed for those
  dates, so their nav is not wrong — only the new label/order differs. The
  calendars self-heal on the next run.

## Out of scope

- Re-rendering historical GH/HN daily pages.
- Any change to story selection, summaries, or page content.
- Email/notification changes beyond appending the AI URL to the existing links
  email when the daily run picks up a new AI edition.
